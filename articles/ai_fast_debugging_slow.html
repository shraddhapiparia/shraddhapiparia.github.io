AI Is Fast. Debugging Is Slow.

The first instinct when using AI for scientific code is always the same: ask for everything at once.

“Write the whole pipeline.”

It feels efficient. One prompt. Ten scripts. Everything done.

In practice, that is usually where the trouble starts.

The code may look complete, but small inconsistencies quickly appear: a file written to the wrong directory, a metadata column renamed in one script but not the next, a reduction called umap in one place and wnn_umap in another. None of these problems are difficult on their own. Together, they make the pipeline fragile.

AI is fast. Debugging is slow.

The goal is not to make AI generate more code. The goal is to give it less room to be wrong.

Start With a Contract, Not Code

Before implementation, define exactly what each script is responsible for.

  • inputs and outputs
  • file names and locations
  • required metadata columns
  • thresholds and assumptions
  • what downstream scripts are allowed to depend on

Once these pieces are fixed, implementation becomes much easier. Before that, every later script is built on moving ground.

For example, if one script always writes a column called pass_qc, every later script can rely on exactly that name. If dimensionality reductions are always saved as pca and lsi, later steps no longer need to guess.

Keep Prompts Narrow

The biggest mistake is asking for the entire pipeline in a single step.

A better pattern is:

  • define the contract for one script
  • implement only that script
  • patch only the specific problem
  • then move to the next stage

Small prompts are not a limitation. They are how you keep the system stable.

Instead of:

Write the entire multi-step pipeline.

Use:

Implement only 03_preprocess_multiome.R from the agreed contract.

Then later:

Patch only these two things. Show the diff only.

Freeze Interfaces Early

Scientific pipelines become much easier when the small details stop changing.

Choose the exact file names, column names, object names, and command-line flags early. Then stop changing them.

Most failures are not complicated. They are small inconsistencies repeated across multiple files.

  • pass_qc vs passed_qc
  • pca vs rna_pca
  • a file written to results/ instead of data/processed/

Each difference seems minor. Together, they create hours of debugging.

Prefer Small Diffs Over Full Rewrites

Once a script exists, the best prompt is usually not “rewrite the whole thing.”

It is:

Patch only this. Show the diff only.

Small diffs are easier to inspect, less expensive, and much less likely to break something that was already working.

This is especially useful for small robustness changes such as path handling, command-line flags, or one missing metadata column.

Test Immediately

Do not wait until the entire pipeline is finished to find out whether script 03 was wrong.

After every script, run the smallest possible test.

  • Does the script run?
  • Does it write the expected file?
  • Do the expected columns exist?
  • Do the object names match what the next step expects?

A bad filename discovered immediately is easy to fix.

The same filename discovered after three later scripts already depend on it is not.

The fastest way to build scientific code with AI is usually the opposite of what feels fast.

Think longer. Specify more. Build smaller.

Then move quickly once the ground is solid.
