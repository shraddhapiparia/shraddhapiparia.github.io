AI Is Fast. Debugging Is Slow.

The first instinct is always the same.

“Write the whole pipeline.”

It feels efficient.

One prompt. Ten scripts. Everything done.

Then you spend the next three hours fixing filenames, column names, paths, object names, assumptions, thresholds, and one flag that silently broke everything downstream.

AI is fast.

Debugging is slow.

The trick is not getting AI to write more code.

The trick is giving it less room to be wrong.

That starts before the code.

Most people prompt for implementation first.

The better move is to start with a contract.

What goes in.

What comes out.

What the files are called.

What columns exist.

What later scripts are allowed to assume.

Once those things are fixed, implementation becomes much easier.

Before that, you are building on moving ground.

The biggest mistake is asking for everything at once.

“Build the full multi-step pipeline.”

That sounds efficient in the same way writing a 40-page plan before starting feels efficient.

It creates the feeling of progress without the reliability of progress.

Better:

Define only the contract for one script.

Then implement only that script.

Then patch only two things.

Then move on.

Small prompts are how you keep the system stable.

The other thing that matters is freezing interfaces early.

Pick the filenames.

Pick the column names.

Pick the object names.

Then stop changing them.

If one script writes pass_qc, every later script should rely on exactly that.

Not passed_qc. Not qc_pass. Not “whatever Claude happened to generate this time.”

Most pipeline failures are not complicated.

They are tiny inconsistencies repeated across multiple files.

One wrong reduction name.

One missing metadata column.

One output written to the wrong directory.

That is why the best prompt after the first implementation is usually not:

“Rewrite the whole thing.”

It is:

“Patch only this. Show the diff only.”

Small diffs are cheaper.

They are easier to inspect.

And they are much less likely to break something that was already working.

The other habit that changed everything for me was testing immediately.

Not after the entire pipeline.

After every script.

Does it run?

Does it write the file?

Does the file contain the columns you expected?

Because the earlier you find an error, the cheaper it is.

A bad filename in script 03 is easy.

A bad filename discovered after scripts 04, 05, and 06 already depend on it is not.

There is a broader lesson here.

AI does not remove the need for engineering discipline.

It increases the cost of not having it.

The fastest way to build scientific code with AI is usually the opposite of what feels fast.

Think longer.

Specify more.

Build smaller.

Then move quickly once the ground is solid.
