How I Use Claude Code for Biotech Workflows

Computational biology is already hard because the underlying biology is complex, noisy, and high stakes. But on top of that, the workflows themselves can become hard to manage. Even when the scientific question is clear, progress often slows because too many moving parts have to stay aligned at once. A Spark join changes one column name. A Nextflow step writes to the wrong path. A single-cell pipeline silently uses the wrong metadata file. By the time the code breaks, the real mistake often happened three steps earlier.

Over the last year, I have started using Claude Code less like autocomplete and more like a junior collaborator: useful, fast, occasionally wrong, and most effective when given structure. The useful pattern is not one giant prompt. It is a system: clear design, persistent memory, explicit constraints, and constant verification.

Why biotech workflows break so easily

Most biotech projects are not one script. They are dozens of scripts connected by assumptions. More often, the harder part is keeping track of how all of the pieces fit together. That is where coding agents become useful: not because they replace expertise, but because they help keep track of the details that are easy to lose.

Start with the design, not the code

Before I ask Claude Code to write anything, I usually sketch the workflow myself. I build the scaffold, write down which scripts should exist, what each script is responsible for, which files move between steps, and what the final outputs should look like. Only after that do I ask the agent to help refine or extend the design. For most projects, I define:

  • the inputs and outputs for each step
  • expected filenames and folder structure
  • required metadata columns
  • how scripts connect to each other
  • what counts as success
  • which failure modes should stop the workflow

Without that design, the agent writes code that looks plausible but drifts away from actual analysis. With a clear design, it becomes much easier to generate scaffolding, build the scripts one step at a time, explore edge cases, and pressure-test parts of the workflow that I may not have thought about.

Depending on the stage of the project, I use Claude Code differently. Early on, I use it for ideation: what scripts are missing, what failure cases could happen, what assumptions might break later. Once the structure is clearer, I use it more as an implementation partner. The more precisely I define the edges of the problem, the more useful the agent becomes, and the less time I spend debugging the middle.

Why memory matters

The first thing that breaks in a long analysis is memory. Not the model's memory. Mine!

By the third day of debugging, I have forgotten which file was the correct one, which comparison already failed, whether the protein list changed, and why I renamed a column in the first place. Turns out, coding agents have the same problem. Short-term memory is useful for the current task. It can keep track of the last few files, commands, and errors. This works well when debugging one issue but projects rarely stay inside one context window. The more useful part is the long-term memory. In practice, I like to think about it in three forms.

  • Episodic memory: what failed before
  • Semantic memory: facts about the project
  • Procedural memory: how the workflow is supposed to run

Most mistakes are not new mistakes, they are old mistakes in slightly different clothing.

Where Claude Code helps most

Claude Code is most useful to me during design and implementation. I use it to turn an idea quickly into a pipeline: which scripts should exist, what each step should produce, what metadata moves through the workflow, and which assumptions or edge cases may fail later. That is especially useful for larger pipelines such as single-cell analysis, proteomics workflows, Nextflow, or Spark analyses. I do use its reasoning traces, but mostly as a monitoring signal rather than a faithful explanation. A confident explanation can still be wrong. One can not evaluate the agent by how convincing its reasoning sounds but more by verification, safety, and the quality of the final outputs.

Finally, the small file that matters most

Before I let Claude Code touch a genomics pipeline or a clinical dataset, I give it a short Claude.md file. Not because I want it to sound smarter, but because I want it to be harder for the agent to sound confident when it is wrong. My version includes rules like:

  • Do not fake expertise. If a task requires domain-specific vaidation that you can't perform, say so upfront and offer to write code or use tools that can.
  • Don't assert runtime behavior, performance, or data outputs. If it can be verified by running code, run it.
  • Push back on requests that have incorrect assumptions rather than building on top of them.
  • State uncertainty explicitly. "I think" and "I'm not sure" are more useful than a confident wrong answer.
  • Ask about the user's context (background, goals, constraints) before giving advice on open-ended questions.
  • Do not use em dashes, use other grammar methods to avoid them.

Those rules matter more in biotech than almost anywhere else. The most useful agent is not the one that always has an answer, it is the one that knows when not to pretend.

Where coding agents still fail

Coding agents are very good at producing code that looks correct which is not the same thing as producing code that is correct. They often invent column names, assume a file exists when it does not, quietly use the wrong group labels, and produce explanations that sound confident but are not actually verified. More often than not, the most dangerous mistakes are the ones that look polished.

Don't trust the code when it runs succesfully, trust it only after manually verifying including but not limited to: do the sample counts match, do the outputs make sense, do the plots look biologically reasonable, do the checks pass, and whether a small hand-checked subset gives the same answer. Reasoning is useful but verification is required.

What I still do myself

The biology still has to come from the scientist. I still decide:

  • what question matters
  • which confounders are important
  • which assumptions are unsafe
  • what counts as evidence
  • when a result is biologically plausible and when it is just noise

Claude Code helps me move faster through the repetitive parts of computational biology: the glue code, the pipeline structure, and the details that are easy to lose. But it does not replace the hard part. The hard part is still deciding what is worth asking in the first place.
